[H1] Query 放在 data 前还是后，如何改变计数学习？
[P] v16.2（query-first）与 v16.3（data-first / query-last）的行为、learning dynamics、attention route 与 hidden-state representation 对比。
[P] 本报告只把证据支持到的层级写成结论；对尚未直接观测的 data→query 聚合明确标记为机制假说。
[H2] 1. 执行摘要
[P] 在同一 seed、相同初始权重、相同数据和相同输出绝对位置下，把 query 从 data 前移动到 data 后，主要改善了 Nonthinking：它更早学会、最终错误更少，并从“答案位置直接广泛扫描 data”转向“答案位置读取紧邻 query”的路由。Thinking 的最终答案已接近天花板，但 trace 生成明显更可靠。
[H3] 本节小结：目前结论与证据缺口
[H4] 目前可以得到的结论
[LI] Query-last 对 Nonthinking 有大而一致的配对提升；Thinking 的答案提升很小，但 trace exact 大幅改善。
[LI] 输出起点始终是 position 262，因此差异不能归因于答案绝对位置变化。
[H4] 欠缺的证据
[LI] 只有一个训练 seed，尚无跨初始化效应分布。
[LI] 尚未直接 probe query-token state 或从 query token 出发的 attention；data→query→answer 目前是受多项间接证据支持的假说。

[H3] 本节小结：目前结论与证据缺口
[H4] 目前可以得到的结论
[LI] Query-last 把 Nonthinking 的高可解码性与近一维有序 centroid geometry 一起提前到 L2；这一点在 probe、几何指标和真实 state 的 3D centroids 中相互一致。
[LI] v16.2 与 v16.3 都不是所有层都保持一条直线：后层会重组几何，因此 full-state ridge R² 与 PC1 geometry 必须分开解读。
[H4] 欠缺的证据
[LI] 线性可解码和 centroid geometry 都不等于模型因果使用；v16.3 尚未完成 centroid steering、state transplant 等与 v16.2 同规格的验证。
[LI] 各 3D panel 的 PCA basis 独立，不能做逐坐标对齐；Thinking answer-state 仍受 count-dependent position 混淆，需要固定位置重测。
[H2] 8. Thinking trace：答案近饱和，但轨迹更稳定
[P] 最终独立 AR suite 中，trace exact 从 81.0% 升到 98.0%，marker recall 从 93.8% 升到 99.1%；最终答案则只从 98.0% 升到 99.0%。这说明 query-last 对 Thinking 的主要可见收益是减少中间 trace 错误，而不是突破已经饱和的 final count。
[P] 删除最后一个 index/marker pair 的 counterfactual 在两边都把 L4 ridge count prediction 约降低 1：v16.2 为 -1.016，v16.3 为 -0.970；gold-vs-(n−1) logit margin 分别变化 -22.91 与 -22.46。因此两种 layout 都继续使用 trace span/边界来形成最终 count readout。
[H3] 本节小结：目前结论与证据缺口
[H4] 目前可以得到的结论
[LI] Query-last 显著提高 Thinking trace exact 与 marker recall，并提高 ordered k-to-k selectivity。
[LI] 两种 layout 对 trace 缩短都表现出约 −1 的内部 count shift，说明核心 span-sensitive readout 仍保留。
[H4] 欠缺的证据
[LI] Thinking final answer 有天花板效应，需要更难 count range 或更长序列才能比较最终能力。
[LI] v16.3 还没有完成与 v16.2 相同的 head ablation、clean-to-corrupt patch 和 residual steering 全套 causal battery。
[H2] 9. 机制综合：为什么 query-last 尤其帮助 Nonthinking？
[P] 数据最支持的解释是“把一个困难的长距离直接 readout，改造成靠近输出的 bottleneck”。v16.3 的 query tokens 位于 data 之后，理论上可以针对自身字符回看完整窗口；答案随后只需从最近的五个 query tokens 读取汇总。这个解释同时预测：direct broad head 变弱、query mass 变强、count representation 提前、Nonthinking 学得更快——四项都被观察到。
[P] 但中间的 Query-token aggregator 仍是虚线：目前没有直接从 query positions 提取 hidden states、attention maps 或做 patch。其他可能解释包括单纯的 RoPE recency 优势、五个 query token 作为短期 key/value cache，或 data 整体平移五位引起的优化变化。
[H3] 本节小结：目前结论与证据缺口
[H4] 目前可以得到的结论
[LI] 四个相互独立的观测——行为、学习速度、fixed-head route、L2 state——共同支持 query bottleneck 解释。
[LI] v16.3 不是把 v16.2 direct broad aggregation 做得更强，而是以不同路线达到更好结果。
[H4] 欠缺的证据
[LI] 没有 query-site probe/patch，因此不能把 data→query 这一步写成已证实机制。
[LI] 当前二点对比无法区分 causal visibility reversal、query recency 和 RoPE distance 三个子因素。
